# Go 学习路线

## 学习目标

这个文档不是通用 Go 教程，而是专门配合 Locator 项目使用的学习路线。目标不是一次性学完语法，而是在做项目的过程中逐步建立四种能力：

1. 能读 Go 代码。
2. 能发现常见 Go 代码错误。
3. 能写简单 Go 代码。
4. 能真正理解 Go 服务的生命周期和 `context.Context`。

## 你最终要达到的能力

### 1. 读代码能力

你需要逐步做到：

- 看懂 `main.go` 启动流程
- 看懂一个 HTTP 请求怎么从 router 进到 handler、service、repository
- 看懂结构体定义、方法接收者、接口声明
- 看懂错误是怎么返回和处理的
- 看懂 `context` 是怎么一路往下传的
- 看懂 goroutine 是在哪里启动、在哪里结束的

### 2. 找错能力

你需要能识别这些常见问题：

- 忘记处理 `err`
- 错误被吞掉，日志信息不足
- 不该用值接收者时用了值接收者
- nil 指针或零值误用
- slice 和 map 初始化错误
- SQL 查询没有超时控制
- goroutine 启动了但没有退出路径
- `context` 没有传递下去
- `context.WithTimeout` 创建了但忘记 `cancel`
- 把 request context 保存到全局或结构体里

### 3. 写简单代码能力

你需要能独立写出：

- 简单结构体
- 简单方法
- 基础 `if`、`for`、`switch`
- JSON 请求和响应结构
- 一个简单 Gin handler
- 一个简单 service 函数
- 一个简单 repository 查询
- 一个带超时的数据库调用
- 一个简单 goroutine 使用场景

### 4. 生命周期和 Context 能力

你需要理解：

- 一个 Go 进程从启动到退出发生了什么
- HTTP 请求的生命周期是什么
- WebSocket 连接的生命周期是什么
- MQTT 客户端的生命周期是什么
- 定时任务的生命周期是什么
- `context` 为什么存在
- `context` 应该由谁创建、谁传播、谁取消
- 超时、取消、优雅退出之间的关系

## 学习方法

后续我们按“项目任务 + Go 讲解”的方式推进，而不是先上完整语法课。

每做一个后端模块，我会同步给你四类信息：

1. 代码在做什么
2. 这里用到了哪些 Go 语法和标准库概念
3. 这里最容易犯什么错
4. 这段代码和生命周期、`context` 有什么关系

## 项目阶段对应学习路线

### 阶段 1：服务骨架

项目任务：

- 搭建 `cmd/server/main.go`
- 初始化配置、日志、数据库、HTTP 服务

你要学的 Go：

- 包和目录结构
- `main` 函数
- 模块和导入
- 变量声明
- 函数和返回值
- `if err != nil`
- 基本 struct

你要重点建立的能力：

- 看懂程序入口
- 看懂初始化顺序
- 看懂依赖如何被组装起来

常见错误：

- 初始化顺序混乱
- 配置未校验
- 错误只打印不返回
- 在 `main` 里堆太多业务逻辑

### 阶段 2：HTTP 和用户登录

项目任务：

- Gin 路由
- 登录接口
- JWT 中间件
- 用户管理接口

你要学的 Go：

- 结构体标签
- JSON 编解码
- 方法接收者
- 接口分层
- 错误处理模式

你要重点建立的能力：

- 看懂一个请求从路由到响应的流转
- 看懂 handler 和 service 的区别
- 看懂输入校验和业务逻辑的边界

常见错误：

- handler 里塞满业务逻辑
- 请求结构和数据库结构混用
- 错误码和业务错误不区分
- 没有对空输入做校验

### 阶段 3：数据库和 repository

项目任务：

- 设备管理
- 用户查询
- GPS 记录入库

你要学的 Go：

- 指针和值
- 方法集
- 错误包装
- 数据访问分层
- `context` 传到 DB 调用

你要重点建立的能力：

- 看懂数据库代码的输入输出
- 看懂为什么 repository 不应该直接处理 HTTP 细节
- 看懂什么时候需要指针

常见错误：

- 查询不带 context
- 把 GORM model 当所有层的通用 DTO
- 返回值和错误语义不清晰
- 忘记判断记录不存在

### 阶段 4：MQTT、WebSocket 和并发

项目任务：

- MQTT 订阅
- GPS 上报处理
- WebSocket 实时推送

你要学的 Go：

- goroutine
- channel 基础
- 并发下的共享状态风险
- 连接管理
- 退出和清理

你要重点建立的能力：

- 看懂并发代码不是“同时跑”这么简单
- 看懂谁负责启动 goroutine
- 看懂谁负责关闭连接和回收资源

常见错误：

- goroutine 泄漏
- 对共享 map 并发读写
- 消息循环没有退出条件
- 出错后不清理连接

### 阶段 5：历史轨迹、围栏和告警

项目任务：

- 轨迹查询
- 围栏判断
- 告警生成

你要学的 Go：

- 纯业务函数设计
- 包边界
- 单元测试思路
- 业务规则与基础设施分离

你要重点建立的能力：

- 看懂“可测试代码”和“框架胶水代码”的区别
- 看懂哪些函数应该纯粹输入输出
- 看懂业务规则如何抽离

常见错误：

- 把数据库查询和算法逻辑写在一起
- 一段逻辑依赖太多外部对象
- 函数副作用过多，难以排查错误

### 阶段 6：服务生命周期和优雅退出

项目任务：

- 定时任务
- 服务关闭
- Docker 部署下的信号处理

你要学的 Go：

- `context.Background`
- `context.WithCancel`
- `context.WithTimeout`
- `signal.NotifyContext`
- `http.Server.Shutdown`

你要重点建立的能力：

- 看懂服务不是只有“启动成功”这一刻
- 看懂一个服务如何正常结束
- 看懂资源为什么必须在退出时被回收

常见错误：

- 收到退出信号后直接 `os.Exit`
- 没有关闭数据库、MQTT、WebSocket 等资源
- 没有等待后台 goroutine 结束
- 优雅退出没有超时边界

## 生命周期专项

这是你要重点啃透的部分。

### 1. 进程生命周期

一个 Go 服务通常经历：

1. 进程启动
2. 读取配置
3. 初始化依赖
4. 启动 HTTP / MQTT / 定时任务
5. 进入运行状态
6. 收到退出信号
7. 停止接收新请求
8. 等待正在处理的任务结束
9. 释放资源
10. 进程退出

你后续读代码时，要习惯问这几个问题：

- 这个对象在哪里创建
- 它什么时候开始工作
- 它什么时候停止
- 它停止时谁负责收尾

### 2. HTTP 请求生命周期

一次 HTTP 请求通常是：

1. 客户端发请求
2. Gin 匹配路由
3. 中间件执行
4. handler 读取参数
5. service 执行业务
6. repository 查询数据库
7. 返回结果
8. 请求结束，request context 失效

重点理解：

- request context 只活在这次请求期间
- 请求结束后不应该继续拿它做后台工作
- 如果客户端断开，请求 context 可能被取消

### 3. WebSocket 生命周期

WebSocket 比普通 HTTP 更长寿：

1. 建立连接
2. 启动读循环和写循环
3. 保持心跳
4. 处理断线
5. 关闭连接
6. 退出相关 goroutine

重点理解：

- 它不是一次请求一来一回
- 它需要显式管理连接对象
- 它更容易出现 goroutine 泄漏

### 4. MQTT 生命周期

MQTT 客户端通常经历：

1. 创建客户端
2. 建立连接
3. 订阅主题
4. 接收消息
5. 断线重连
6. 服务退出时主动关闭

重点理解：

- 连接不是永远稳定的
- 消息回调通常是长期存在的
- 重连和关闭都要有明确策略

## Context 专项

### `context` 是什么

`context.Context` 是一条请求或任务链路上的控制信号和元数据容器。它主要解决四件事：

- 取消
- 超时
- 截止时间
- 跨调用链传递少量请求级数据

### 什么时候用 context

适合使用 `context` 的场景：

- HTTP 请求调用 service
- service 调用 repository
- 数据库查询
- 外部 RPC 或 HTTP 调用
- 需要超时控制的任务
- 服务关闭时通知后台 goroutine 停止

### 什么时候不要滥用 context

不应该这样做：

- 不要把 `context` 存在结构体字段里长期复用
- 不要把业务参数塞进 `context`
- 不要用 `context` 代替正常的函数参数
- 不要创建了 timeout context 却不 cancel

### 你必须理解的几种 context

#### `context.Background()`

一般作为顶层根 context，常用于程序启动期。

#### `context.TODO()`

临时占位用，说明这里未来要补上正确 context。项目代码里应尽量少用。

#### `context.WithCancel(parent)`

用于手动取消子任务。

#### `context.WithTimeout(parent, d)`

用于给任务设置超时边界。

#### `context.WithDeadline(parent, t)`

用于设置明确截止时间。

## 读 Go 代码时的固定检查清单

以后你看到一段 Go 代码，优先按这个顺序检查：

1. 这个函数的输入是什么，输出是什么。
2. 错误在哪里返回，是否被处理。
3. 有没有 `context.Context`，它从哪里来，传到哪里去。
4. 有没有 goroutine，谁负责让它退出。
5. 有没有外部资源，例如 DB、MQTT、WebSocket、文件句柄，谁负责关闭。
6. 这个函数属于哪一层，是否越层做了不该做的事。

## 发现 Go 代码错误时的思路

以后排错尽量按这个顺序：

1. 先确认现象：编译错误、运行错误、逻辑错误还是并发问题。
2. 看函数签名和返回值。
3. 看 `err` 是否被忽略。
4. 看 nil 值和零值是否合理。
5. 看 `context` 是否缺失或提前取消。
6. 看 goroutine 是否没有退出。
7. 看日志是否能串起整条调用链。
